# CI/CD

### Логика развёртывания:
0. Установка необходимых для работы пакетов __Sourcecraft__.
1. Настройка SSH и known hosts.
* При возникновении ошибки, в последующих шагах создаётся файл ```failure.reason``` с именем шага, на котором возникла ошибка.
2. Синхронизация файлов с репозиторием (rsync).
   * Исключаются файлы: 
      * ```.git```;
      * ```.gitignore```;
      * ```.sourcecraft```;
      * ```.env.example```;
      * ```README.md```.
* Последующие шаги происходят через shh сессии.
3. Настройка ```.env```.
   1. Файл ```.env``` заполняется с помощью функции ```update_env_var```.
   2. Пример использования: ```update_env_var "<variable name>" "<value>"```.
4. Остановка существующего контейнера и запуск нового.
5. Если развертывание прошло успешно то:
   1. Создаем новый бэкап.
   2. Удаляем старый бэкап.
6. При возникновении ошибки:
   1. Удаляем новый бэкап.
   2. Откатываемся к предыдущей версии.
   3. На сервер отправляется файл с причиной ошибки ```failure.reason```.
7. Удаление файлов конфигурации ```nginx-proxy``` и ```.env```



# portal-proxy

## Функционал системы
Разработанное решение реализует следующие функции:

1. **Приём входящих HTTP и HTTPS-запросов** на сервере с публичным IP-адресом.
2. **Автоматическое получение и обновление TLS-сертификатов** через Let's Encrypt.
3. **Проксирование запросов на внешние HTTPS-ресурсы**, расположенные на других серверах.
4. **Поддержка нескольких доменов в рамках одного docker-compose проекта.**
5. **Раздельное логирование** для каждого домена.
6. **Централизованная конфигурация параметров Nginx**, включая заголовки безопасности, настройки соединений и формат логирования.

Система легко масштабируется и позволяет добавлять новые домены и upstream-серверы без изменения основной архитектуры.


## Архитектура системы
Архитектура системы построена по принципу обратного прокси-сервера.

Схема взаимодействия компонентов:

```
Клиент
   ↓ HTTPS
nginx-proxy (Docker)
   ↓ HTTPS
Удалённый сервер (upstream)
```



## Основные компоненты системы

### Контейнер nginx-proxy

Контейнер nginx-proxy выполняет роль основного обратного прокси-сервера. Он автоматически формирует конфигурацию Nginx на основе информации о контейнерах Docker и их переменных окружения. 

Основные функции:

* обработка HTTP/HTTPS-соединений;
* проксирование запросов на upstream-серверы;
* применение общих настроек безопасности и логирования.

#### Конфигурация
Конфигурация nginx-proxy разделена на несколько файлов для логического разделения настроек. Такое решение позволяет упростить сопровождение системы и повторное использование параметров.

Файлы конфигурации используемые для задания общих параметров:

* **custom-global.conf** — основные параметры сервера;
* **logging.conf** — формат и правила логирования;
* **headers.conf** — заголовки безопасности HTTP;
* **gzip.conf** — настройки сжатия.


#### Конфигурация доменов (директория vhost.d)

Для каждого домена создаётся отдельный файл конфигурации.

Он содержит:

* настройки TLS;
* индивидуальные файлы логирования;
* правила проксирования на удалённый сервер;
* параметры тайм-аутов соединений.

#### Логирование

Система реализует два уровня логирования:

1. **Общее логирование сервера** — фиксирует события работы Nginx.
2. **Отдельные логи для каждого домена** — позволяют анализировать трафик конкретного ресурса.

Все журналы сохраняются в отдельный Docker-том, что упрощает их анализ и резервное копирование.


### Контейнер acme-companion
Контейнер acme-companion отвечает за автоматическую работу с TLS-сертификатами:

* выпуск сертификатов;
* их обновление;
* размещение сертификатов в общем томе Docker.

Этот компонент взаимодействует с nginx-proxy и автоматически определяет домены, для которых требуется выпуск сертификата.


### Контейнер(ы)-триггеры доменов

Для каждого домена создаётся отдельный контейнер-триггер. Он не обрабатывает сетевой трафик и используется исключительно для декларативного описания домена.

Через переменные окружения задаются:

* доменное имя;
* параметры выпуска сертификата.

Эти данные используются nginx-proxy для автоматического создания виртуального хоста.



В результате работы была разработана контейнеризированная система обратного проксирования, позволяющая:

* обслуживать несколько доменов;
* автоматически выпускать TLS-сертификаты;
* проксировать трафик на внешние HTTPS-серверы;
* вести раздельное логирование для каждого домена;
* масштабировать систему без изменения основной архитектуры.

Разработанное решение может использоваться в качестве универсальной точки входа для веб-ресурсов, размещённых на различных серверах.
